<objective>
Wire the CLI entry point, URL resolver, and batch processing loop — making the pipeline invocable as `yt-to-skill <url>` for single videos, playlists, and channels.

Purpose: This is the user-facing interface. After this plan, the full pipeline is usable end-to-end from the command line. URL resolution handles single/playlist/channel detection. The batch loop processes videos sequentially with per-video error isolation. Progress lines and a summary table give feedback.

Output: `yt_to_skill/resolver.py`, `yt_to_skill/cli.py`, extended `config.py` and `orchestrator.py`, `pyproject.toml` entry point, plus tests.

  </behavior>
  <action>
    **Step 1: Extend `yt_to_skill/config.py`**

    Add `skills_dir: Path = Path("skills")` field to `PipelineConfig`. This makes the skill output directory configurable via env var (`SKILLS_DIR`) or CLI flag. Do NOT add `force` to config (CLI-only per research recommendation).

    **Step 2: Extend `yt_to_skill/orchestrator.py`**

    Add a 6th stage block after the extract stage that calls `run_skill(video_id, config.work_dir, config.skills_dir)`. Import `run_skill` from `yt_to_skill.stages.skill`. Follow the same try/except pattern as existing stages. The skill stage should NOT return early on error — it's the last stage, just append the StageResult.

    Also add a `force` parameter to `run_pipeline`: `def run_pipeline(video_id: str, config: PipelineConfig, *, force: bool = False) -> list[StageResult]`. Pass `force=force` to `run_skill()`.

    **Step 3: Create `yt_to_skill/cli.py`**

    `main() -> None`:
    1. Parse args with argparse:
       - `url` (positional): YouTube video, playlist, or channel URL
       - `--output-dir` (default: "skills"): Base directory for skill output
       - `--force` (store_true): Reprocess even if SKILL.md exists
       - `--verbose` (store_true): Show full logs
    2. Configure loguru: if not verbose, set level to "INFO" and use a minimal format showing just stage progress lines. If verbose, use default loguru format with DEBUG level.
    3. Create PipelineConfig. Override `skills_dir` from `--output-dir` if provided.
    4. Call `resolve_urls(args.url)` to get list of video IDs. Wrap in try/except for NetworkError — print error and exit(1).
    5. Batch loop: iterate over video_ids. For each:
       a. Print progress: video ID being processed
       b. Call `run_pipeline(video_id, config, force=args.force)` inside try/except
       c. On success: check StageResults for errors. Record status as success or partial.
       d. On `SkillError`: record error category + message. Print inline warning. Continue to next video.
       e. On other exceptions: record as "UNKNOWN" error. Continue to next video.
       f. Track results: list of `(video_id, status, detail)` tuples.
    6. Default progress display: one line per stage as it completes: `print(f"  ✓ {stage_name}")` for completed stages, `print(f"  ✗ {stage_name}: {error}")` for failed stages. To achieve this, check each StageResult after run_pipeline returns and print accordingly.
    7. Handle non-strategy videos: if filter stage returned and video was filtered as non-strategy (check StageResults — filter is last result and pipeline returned early), show as "⚠ skipped — not a strategy video" in summary. This is NOT a failure — does not affect exit code.
    8. Summary table (always for batch, i.e., >1 video): print formatted table with columns: Video ID, Status (✓/✗/⚠), Detail (skill path or error). Truncate video IDs/details if too long. Use simple f-string alignment.
    9. Exit: `sys.exit(0)` if all succeeded or skipped, `sys.exit(1)` if any failed.

    The entire batch loop MUST be wrapped in try/finally to ensure the summary table prints even on unexpected errors.

    **Step 4: Register CLI in `pyproject.toml`**

    Add `[project.scripts]` section: `yt-to-skill = "yt_to_skill.cli:main"`

    **Step 5: Write tests in `tests/test_cli.py`**

    Mock `resolve_urls`, `run_pipeline`, and `run_skill` (or just `run_pipeline` since it now includes skill stage). Use `unittest.mock.patch` and `capsys` for stdout capture. Test:
    - Single video: mock resolve_urls to return ["vid1"], mock run_pipeline to return success StageResults. Assert exit code 0.
    - Output dir: assert PipelineConfig receives overridden skills_dir.
    - Force flag: assert run_pipeline receives force=True.
    - Batch failure: mock resolve_urls to return ["vid1", "vid2"]. First call succeeds, second raises SkillError. Assert vid1 processed, exit code 1.
    - Summary table: capture stdout, verify table contains video IDs and status markers.
    - Non-strategy: mock run_pipeline to return results where filter stage is last and video was filtered. Assert exit code 0, summary shows skipped.

    Use `pytest` with `monkeypatch` for sys.argv. Use `SystemExit` catch for exit code testing.

    Also add a test for PipelineConfig.skills_dir default in test_cli.py or test_config.py.
